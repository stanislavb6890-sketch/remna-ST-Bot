# План интеграции sing-box нод в CLOAKNET

## 1. Обзор и цели

**Идея:** Админ добавляет sing-box ноды через готовый Docker Compose на своих серверах. В панели видит ноды (онлайн, нагрузка, кол-во пользователей). Создаёт категории и тарифы «sing-box» (например: «1 доступ 30 дней», «5 доступов 30 дней»). Клиенты покупают в боте/кабинете и получают доступы в виде подписочной ссылки (subscription URL) или клиентского конфига (VLESS / Shadowsocks / Trojan / Hysteria2) с ограничениями по трафику и сроку действия.

**Отличие от прокси:** Прокси — это логин:пароль на SOCKS/HTTP. Sing-box — это VPN-нода с протоколами VLESS, Shadowsocks, Trojan, Hysteria2 и др.; один «слот» = один пользователь (UUID или пара логин/пароль) на инбаунде; клиент получает ссылку на подписку или готовый конфиг для приложения (v2rayN, Nekoray, sing-box client, Hysteria2 client и т.д.).

**Ключевые требования:**
- Админ: «Добавить sing-box ноду» → получает готовый `docker-compose` → поднимает ноду → нода отображается в панели (онлайн, метрики).
- Админ: категории и тарифы sing-box (N доступов на M дней, цена).
- Покупка в боте/кабинете → выдача подписочной ссылки или конфига на каждый «слот».
- Ограничения на слот: трафик (байты), срок действия.
- Мониторинг: по ноде и по слотам (трафик, подключения — по возможности).

- **Панель (API + фронт):** управление нодами, категориями, тарифами; при оплате — создание слотов, привязка к ноде.
- **Нода:** один Docker-образ с агентом + sing-box. Агент регистрируется по токену, шлёт heartbeat и метрики, тянет список слотов (пользователей) и обновляет конфиг sing-box (JSON) или использует SSM API (для Shadowsocks managed), перезапускает/перезагружает sing-box.

**Рекомендация:** Расширить существующую `Payment`: добавить `singboxTariffId` (nullable). После успешной оплаты: если `singboxTariffId` задан — создать `slotCount` слотов, распределить по нодам (round-robin или по наименьшей загрузке), сгенерировать UUID/логин и секрет для каждого слота.

### 3.6. Разные конфиги по протоколам — что делать

У каждого протокола свой формат инбаунда в sing-box и своя ссылка для клиента. Ошибок не будет, если **протокол зафиксирован на ноде**, а не на тарифе.

**Принцип: одна нода = один протокол.**

- В БД у ноды хранится `protocol` (VLESS, SHADOWSOCKS, TROJAN, HYSTERIA2). Слот привязан к ноде → у слота по сути тот же протокол (берём из ноды).
- При создании слотов слоты выдаются только на ноды с подходящим протоколом (если тариф привязан к типу протокола — см. фазу 4) или на любые ноды, а протокол слота = протокол ноды.

**На ноде (агент):**

- Агент при `GET /slots` получает от API и протокол ноды (в ответе ноды или в метаданных слотов). По одному значению `protocol` выбирается **один** генератор конфига:
  - **VLESS** → блок `inbounds` с `type: "vless"`, `users: [ { name, uuid } ]`.
  - **SHADOWSOCKS** → блок с `type: "shadowsocks"`, `method`, `users: [ { name, password } ]` или SSM API.
  - **TROJAN** → блок с `type: "trojan"`, `users: [ { name, password } ]`.
  - **HYSTERIA2** → блок с `type: "hysteria2"`, `users: [ { name, password } ]`, `tls`, опционально `obfs` и т.д.
- Реализация: одна функция «собрать конфиг sing-box» с веткой `switch (protocol)` или отдельная функция на протокол (например `buildVlessInbound(slots)`, `buildHysteria2Inbound(slots)`), общий шаблон (route, outbounds) один и тот же.

**В панели (выдача клиенту):**

- При выдаче подписки или одной ссылки для слота у нас есть: `node.protocol`, `node.publicHost`, `node.port`, `node.tlsEnabled`, `slot.userIdentifier`, `slot.secret`.
- По `node.protocol` выбирается **форматер ссылки** (один на протокол):
  - VLESS → `vless://uuid@host:port?...`
  - SHADOWSOCKS → `ss://base64(...)@host:port#...`
  - TROJAN → `trojan://password@host:port?...`
  - HYSTERIA2 → `hysteria2://...` / `hy2://...`
- Реализация: на бэкенде один метод «сгенерировать ссылку для слота», внутри `switch (node.protocol)` или объект `linkFormatters[node.protocol](node, slot)`.

**Итог:** не нужно «универсального» конфига. Нужно: (1) хранить протокол на ноде; (2) в агенте — один шаблон конфига на протокол (или один переключатель); (3) в API выдачи — один форматер ссылки на протокол. Добавление нового протокола = новая ветка в агенте и новый форматер в панели.

### 3.7. Редактирование конфига в админке (JSON-редактор)

Чтобы можно было менять конфиг sing-box прямо из админки (порты, TLS, route, outbounds, обфускация и т.д.), в ноде хранится **кастомный конфиг** — поле `customConfigJson` (текст, валидный JSON).

**Поведение:**

- Если `customConfigJson` **задан** — агент использует его как основу. В конфиге должен быть ровно один инбаунд с фиксированным тегом (например `"tag": "CLOAKNET-in"`). Агент **подменяет** у этого инбаунда массив `users` на список из слотов (из API), остальное не трогает. Итоговый JSON пишется на диск и sing-box перезапускается/перезагружается.
- Если `customConfigJson` **пуст** — агент генерирует конфиг целиком по протоколу ноды и шаблону (как в п. 3.6 и 5.2).

**Договорённость:** инбаунд, в который подставляются пользователи из панели, должен иметь тег `CLOAKNET-in`. Админ в редакторе может менять что угодно (listen, tls, obfs, route, outbounds, log), но этот тег у «управляемого» инбаунда нужно сохранять.

**Админка:**

- В карточке/детали ноды — кнопка «Редактировать конфиг» (или вкладка «Конфиг»).
- Открывается **JSON-редактор** (textarea с подсветкой синтаксиса или готовый компонент вроде Monaco/CodeMirror в режиме JSON). Показывается текущий `customConfigJson` или подставленный дефолтный шаблон по протоколу ноды (чтобы админ мог сразу править).
- Кнопки: «Сохранить», «Сбросить к шаблону по протоколу», «Проверить JSON». При сохранении — валидация JSON на бэкенде, запись в `SingboxNode.customConfigJson`. Агент при следующем запросе получит новый конфиг и применит его на ноде.

**API:**

- `GET /admin/singbox-nodes/:id` — в ответе поле `customConfigJson` (если нужно отдельно: `GET /admin/singbox-nodes/:id/config`).
- `PATCH /admin/singbox-nodes/:id` с телом `{ "customConfigJson": "..." }` или `PUT /admin/singbox-nodes/:id/config` с телом `{ "config": { ... } }` (JSON). Сохранение как строка в БД.
- Для агента: в ответе `GET /api/singbox-nodes/:id/slots` добавить поле `customConfigJson` (или отдельный `GET /api/singbox-nodes/:id/config`), чтобы агент мог брать кастомный конфиг и подставлять в него `users` из того же ответа.

**Валидация:** при сохранении проверять, что это валидный JSON и при наличии инбаунда с тегом `CLOAKNET-in` — структура подходит (есть поле `users` или агент его перезапишет). Опционально: базовая проверка схемы sing-box (log, inbounds, outbounds).

## 4. Сценарии

### 4.1. Админ: добавление sing-box ноды

1. В админке раздел «Sing-box» → «Ноды» → кнопка «Добавить ноду».
2. Ввод названия, выбор протокола (VLESS / Shadowsocks / Trojan / Hysteria2), порт(ы), TLS (да/нет; для Hysteria2 TLS обязателен).
3. Генерация уникального `token`. Показ готового блока:
   - `docker-compose.yml` с переменными `CLOAKNET_API_URL`, `SINGBOX_NODE_TOKEN`, протокол, порт.
   - Краткая инструкция: скопировать, на сервере `docker compose up -d`.
4. После запуска контейнера:
   - Агент отправляет `POST /api/singbox-nodes/register` с телом `{ "name": "...", "protocol": "vless", "port": 443, "tlsEnabled": true }`.
   - API проверяет токен, создаёт/обновляет SingboxNode (status = ONLINE, lastSeenAt, publicHost).
   - Ответ: `{ "nodeId": "...", "pollIntervalSec": 60 }`.
5. В списке нод админ видит новую ноду (онлайн, 0 слотов, 0 трафика).

### 4.2. Админ: heartbeat и метрики

1. Агент раз в N секунд: `POST /api/singbox-nodes/:id/heartbeat` с телом:
   - `connections` — общее кол-во соединений (если есть, иначе 0),
   - `trafficIn`, `trafficOut` — общий трафик ноды,
   - `slots`: `[ { "slotId": "...", "trafficUsed": 0, "connections": 0 }, ... ]`.
2. API обновляет SingboxNode и SingboxSlot (trafficUsedBytes, currentConnections).
3. Если heartbeat не приходит дольше порога (например 5 минут) — нода помечается OFFLINE.

### 4.3. Админ: категории и тарифы sing-box

1. «Категории sing-box»: CRUD (название, порядок).
2. «Тарифы sing-box»: внутри категории — название, кол-во доступов (slotCount), срок (durationDays), лимит трафика на слот, цена, валюта.
3. Список тарифов с возможностью редактирования и отключения (enabled).

### 4.4. Админ: просмотр нод и слотов

1. Список нод: таблица (название, протокол, статус, lastSeenAt, кол-во слотов, трафик).
2. Клик по ноде → детали: список слотов (клиент, userIdentifier, срок, трафик использовано/лимит).
3. Опционально: графики по трафику/нагрузке.

### 4.4a. Админ: редактирование конфига ноды (JSON)

1. В детали ноды — кнопка или вкладка «Редактировать конфиг».
2. Открывается JSON-редактор с текущим кастомным конфигом (или шаблоном по протоколу). Админ правит JSON (порты, TLS, route, outbounds, obfs и т.д.), сохраняет.
3. Бэкенд валидирует JSON, сохраняет в `customConfigJson`. При следующем цикле агент подтянет конфиг, подставит в инбаунд с тегом `CLOAKNET-in` пользователей из слотов и применит конфиг на ноде.

### 4.5. Клиент: покупка в боте/кабинете

1. В боте/кабинете раздел «Sing-box» (или «Доступы») — категории и тарифы.
2. Выбор тарифа, способ оплаты: баланс, Platega, ЮMoney, ЮKassa (аналогично VPN и прокси).
3. После успешной оплаты:
   - Бэкенд создаёт `slotCount` слотов: выбор нод (ONLINE), распределение, генерация UUID (VLESS/Trojan) или логин+пароль (Shadowsocks, Hysteria2) для каждого слота.
   - Выдача клиенту:
     - **Подписочная ссылка:** `https://panel.example.com/api/client/singbox/subscription/:clientId/:token` (токен подписки на уровне клиента или на слот).
     - Или отдельная ссылка на слот: `https://panel.example.com/api/client/singbox/slot/:slotId/subscription?token=...`.
     - Либо готовый клиентский конфиг (JSON для sing-box / v2rayN) в виде скачивания или копирования.
4. Раздел «Мои доступы»: список активных слотов, кнопка «Скопировать ссылку подписки» / «Скачать конфиг», срок окончания, использовано трафика / лимит.

### 4.6. Выдача формата для клиента

- **Subscription URL:** один URL на клиента, возвращающий список нод и пользователей в формате Base64 (стандарт subscription для v2ray/Clash). Или один URL на слот — одна строка конфига.
- **Конфиг:** для каждого слота протокол берётся из ноды (см. **3.6**); по протоколу выбирается форматер. На основе Node (host, port, protocol, tls) и Slot (userIdentifier, secret) генерируется строка в формате:
  - VLESS: `vless://uuid@host:port?type=tcp&security=tls#name`
  - Shadowsocks: `ss://base64(method:password)@host:port#name`
  - Trojan: `trojan://password@host:port?security=tls#name`
  - Hysteria2: `hysteria2://[user:][password@]host:port?insecure=0&sni=...#name` (по [URI Scheme](https://v2.hysteria.network/docs/developers/URI-Scheme); для sing-box инбаунда — пользователи с полем `password`, при необходимости `name:password` в auth).
- Подписочный эндпоинт на бэкенде отдаёт текст (один конфиг на строку или Base64 бандл) для подстановки в приложение клиента.

## 5. Node Agent и sing-box на ноде

### 5.1. Задачи агента

1. При старте: регистрация по токену (`POST /api/singbox-nodes/register`).
2. Периодически: heartbeat с метриками (`POST /api/singbox-nodes/:id/heartbeat`).
3. Периодически: запрос списка слотов для этой ноды (`GET /api/singbox-nodes/:id/slots`) — userIdentifier, secret, expiry, trafficLimit; при необходимости отдельно или в том же ответе — кастомный конфиг ноды (`customConfigJson`).
4. По списку слотов формировать конфиг sing-box:
   - **Если у ноды задан `customConfigJson`** (см. п. **3.7**): взять этот JSON, найти инбаунд с тегом `CLOAKNET-in`, подставить в него массив `users` из слотов (в формате по протоколу ноды), записать итоговый конфиг на диск, перезапуск sing-box.
   - **Если кастомного конфига нет** (см. п. **3.6**): один протокол на ноду, свой генератор конфига на протокол:
     - **Вариант A:** полная пересборка JSON-конфига: общий шаблон (listen, outbounds, route) + блок `inbounds` по протоколу ноды (VLESS/SS/Trojan/Hysteria2) с массивом `users` из слотов; перезапуск sing-box или SIGHUP (если поддерживается).
     - **Вариант B (только Shadowsocks):** использовать [SSM API](https://sing-box.sagernet.org/configuration/service/ssm-api/) sing-box: инбаунд с `managed: true`, агент добавляет/удаляет пользователей через SSM API, без перезапуска.
5. Сбор статистики: по логам sing-box, или [V2Ray API](https://sing-box.sagernet.org/configuration/experimental/v2ray-api/) (gRPC, per-user stats), или SSM API cache — и отправка в heartbeat.

### 5.2. Выбор протокола и конфигурации ноды

- **VLESS:** популярен, один UUID на пользователя, легко генерировать ссылки. Конфиг sing-box: inbound типа `vless` с массивом `users` (name, uuid). При изменении слотов — переписать конфиг и перезапустить sing-box (или hot-reload, если реализован).
- **Shadowsocks:** можно управлять динамически через SSM API (sing-box 1.12+), без перезапуска. Иначе — как VLESS, перезапись конфига.
- **Trojan:** по структуре близко к VLESS (users с password), перезапись конфига при изменении слотов.
- **Hysteria2:** инбаунд `type: "hysteria2"` в sing-box (1.11+); у каждого пользователя `name` и `password`. TLS обязателен. Опционально: `up_mbps`/`down_mbps`, обфускация Salamander (`obfs`), masquerade. При изменении слотов — перезапись конфига и перезапуск sing-box. Выдача клиенту: ссылка `hysteria2://` или `hy2://` (auth: пароль или `name:password`).

Рекомендация для первого этапа: **VLESS** — простой конфиг, одна ссылка на слот, предсказуемая выдача. Опционально: Shadowsocks с SSM API, **Hysteria2** — хорошая устойчивость к помехам и скорость (QUIC), востребован у клиентов.

### 5.3. Структура Docker-образа ноды

- Базовый образ: node:20-alpine или go (если агент на Go).
- Установка sing-box (бинарник из релизов или пакет).
- Скрипт агента: регистрация, heartbeat, загрузка слотов, генерация конфига sing-box (JSON), перезапуск sing-box (или вызов API).
- Entrypoint: запуск агента и sing-box (два процесса в контейнере или supervisord).
- Переменные окружения: `CLOAKNET_API_URL`, `SINGBOX_NODE_TOKEN`, опционально `PROTOCOL`, `PORT`, `TLS_ENABLED`.

### 5.4. API для агента (защита)

- Все запросы агента: заголовок `X-Singbox-Node-Token: <token>`.
- Регистрация только по токену, заранее созданному в админке. После регистрации — привязка nodeId к токену; в heartbeat и getSlots проверять соответствие nodeId и токена.

### 5.5. Конфиг из админки (customConfigJson)

- В ответе `GET /api/singbox-nodes/:id/slots` (или в отдельном `GET /api/singbox-nodes/:id/config`) API отдаёт поле `customConfigJson` ноды, если оно задано.
- Агент: если `customConfigJson` есть — парсит JSON, находит в `inbounds` элемент с `tag === "CLOAKNET-in"`, подменяет у него `users` на массив, собранный из слотов (в формате выбранного протокола: VLESS — `{ name, uuid }`, SS/Hy2 — `{ name, password }` и т.д.), записывает итоговый JSON в файл конфига sing-box и перезапускает процесс. Если `customConfigJson` нет — строит конфиг только из шаблона по протоколу (п. 5.1–5.2).

## 6. Безопасность и ограничения

- Токен ноды хранить в БД (или хэш). Выдача слотов только на ноды ONLINE и не DISABLED.
- UUID/секреты слотов: генерировать криптостойко. В API для агента отдавать секрет только для своего nodeId.
- Подписочный URL для клиента защитить токеном (например одноразовый или привязанный к clientId + слоту), чтобы по ссылке не мог получить конфиг чужой.
- Rate limit на регистрацию и heartbeat. При отзыве доступа: статус слота REVOKED; агент при следующем getSlots исключает этот слот из конфига.

## 7. Этапы реализации

### Фаза 1: Бэкенд и админка (ноды, без продаж)

1. Миграции: таблицы SingboxNode, SingboxCategory, SingboxTariff, SingboxSlot; в Payment добавить `singboxTariffId`.
2. API: CRUD нод (создание записи с токеном при «Добавить ноду»), генерация токена.
3. API для агента: `POST /api/singbox-nodes/register`, `POST /api/singbox-nodes/:id/heartbeat`, `GET /api/singbox-nodes/:id/slots` (возвращает слоты: userIdentifier, secret, expiresAt, trafficLimitBytes; и при наличии — `customConfigJson` ноды для редактора из админки). Админ API: чтение/запись `customConfigJson` ноды (GET/PATCH ноды или отдельный эндпоинт config).
4. Админка: раздел «Sing-box» → вкладка «Ноды»: кнопка «Добавить ноду», вывод docker-compose + инструкция; список нод (статус, lastSeen, метрики). Детали ноды — список слотов. Редактор конфига ноды (JSON) — см. п. 3.7: кнопка «Редактировать конфиг», сохранение `customConfigJson`.
5. Контейнер-агент + sing-box: образ (например VLESS inbound), скрипт регистрации/heartbeat/getSlots, генерация JSON-конфига sing-box, перезапуск sing-box.

### Фаза 2: Категории, тарифы, оплата и слоты

1. Админка: CRUD категорий и тарифов sing-box (slotCount, durationDays, trafficLimit, price).
2. Backend: при оплате (balance или webhook Platega/YooMoney/YooKassa) проверка `singboxTariffId` → создание slotCount слотов, выбор нод (ONLINE), распределение, генерация UUID (или логин/пароль для SS) на каждый слот.
3. Выдача в боте: после оплаты — сообщение с подписочной ссылкой или ссылкой на скачивание конфига для каждого слота (или одна общая подписка на клиента).
4. Бот: раздел «Sing-box» — список тарифов, выбор и оплата; раздел «Мои доступы» — активные слоты, копирование subscription URL / конфига.

### Фаза 3: Кабинет и доработки

1. Кабинет: раздел «Sing-box» — те же тарифы и «Мои доступы» с кнопкой «Скопировать ссылку подписки» / «Скачать конфиг».
2. Эндпоинт подписки: `GET /api/client/singbox/subscription?token=...` или по slotId — отдаёт текст (Base64 или построчно конфиги) для подстановки в приложение.
3. Обновление трафика по heartbeat; отображение в админке и у клиента («использовано X из Y ГБ»).
4. Автоотключение слота при истечении срока или превышении лимита трафика; уведомления в боте/email по желанию.

### Фаза 4 (опционально)

- Поддержка нескольких протоколов на одной ноде (VLESS + Shadowsocks, Hysteria2 и т.д.) и выбор протокола в тарифе.
- Управление слотом в админке: смена пароля (SS / Trojan / Hysteria2), отзыв доступа, продление.
- Резервирование нод по гео/тегам (как у прокси).
- Графики нагрузки и трафика по нодам, экспорт слотов в CSV.

## 8. Краткий чеклист по формату выдачи

- Один слот = один пользователь на одной ноде (один UUID для VLESS/Trojan; один пользователь с паролем для Shadowsocks или Hysteria2).
- Клиенту выдаём:
  - Подписочную ссылку (одну на слот или одну на клиента со всеми слотами).
  - Или готовые строки: `vless://...`, `ss://...`, `trojan://...`, `hysteria2://...` / `hy2://...` для вставки в приложение.
- В «Мои доступы» показывать: срок действия, лимит трафика (если задан), использовано трафика.

## 9. Сравнение с интеграцией прокси

| Аспект | Прокси (3proxy) | Sing-box |
|--------|------------------|----------|
| Протокол | SOCKS5, HTTP | VLESS, Shadowsocks, Trojan, Hysteria2 |
| Единица выдачи | логин:пароль | UUID или логин+пароль |
| Формат клиенту | socks5://login:pass@host:port | subscription URL или vless://... / ss://... |
| Конфиг на ноде | Файл 3proxy | JSON sing-box |
| Динамическое управление | Перезапись конфига + перезапуск | Перезапись конфига или SSM API (SS) |
| Трафик по пользователям | Логи 3proxy | V2Ray API / логи / SSM cache |

План можно использовать как ТЗ для реализации по фазам. Следующий шаг — детализация API (пути, тела запросов/ответов) и пример docker-compose для sing-box ноды.
