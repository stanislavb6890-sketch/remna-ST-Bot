# План интеграции продажи прокси в STEALTHNET

## 1. Обзор и цели

**Идея:** Админ добавляет прокси-ноды через готовый Docker Compose на своих серверах. В панели видит ноды (онлайн, нагрузка, кол-во пользователей). Создаёт категории и тарифы прокси (например: «1 прокси 30 дней», «5 прокси 30 дней»). Клиенты покупают в боте/кабинете и получают индивидуальные доступы в формате `protocol:login:pass:ip:port` (HTTP и SOCKS5), с ограничениями по трафику и кол-ву подключений.

**Ключевые требования:**
- Админ: «Добавить прокси» → получает готовый `docker-compose` → поднимает ноду (SOCKS5 + HTTP) → нода отображается в панели (онлайн, метрики).
- Админ: категории прокси и тарифы (N прокси на M дней, цена).
- Покупка в боте/кабинете → выдача индивидуальных доступов (логин:пароль на каждую «прокси»).
- Ограничения на слот: трафик (байты), кол-во одновременных подключений, срок действия.
- Мониторинг: по ноде и по слотам (трафик, подключения).

## 2. Архитектура

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         STEALTHNET (существующее)                        │
├─────────────────────────────────────────────────────────────────────────┤
│  Админ-панель     │  Новое: Прокси → Ноды, Категории, Тарифы, Статистика │
│  Бот / Кабинет    │  Новое: Раздел «Прокси» → выбор тарифа → оплата      │
│  API              │  Новое: CRUD нод, тарифы, слоты; API для агента     │
├─────────────────────────────────────────────────────────────────────────┤
│  PostgreSQL       │  Новое: ProxyNode, ProxyCategory, ProxyTariff,     │
│                    │         ProxySlot, ProxyPayment (или расширить Payment)│
└─────────────────────────────────────────────────────────────────────────┘
                                    │
                    HTTPS (токен ноды в заголовке)
                                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  Прокси-нода (сервер админа)                                             │
│  ┌─────────────────┐  ┌──────────────────┐  ┌─────────────────────────┐ │
│  │  Node Agent     │  │  3proxy / gost   │  │  Метрики (трафик,        │ │
│  │  - регистрация  │──│  SOCKS5 + HTTP   │  │  подключения по юзерам) │ │
│  │  - heartbeat    │  │  auth из API     │  │  → отправка в API       │ │
│  │  - загрузка     │  │  лимиты трафика  │  └─────────────────────────┘ │
│  │    списка юзеров│  │  и соединений    │                              │
│  └─────────────────┘  └──────────────────┘                              │
└─────────────────────────────────────────────────────────────────────────┘
```

- **Панель (API + фронт):** управление нодами, категориями, тарифами; при оплате — создание слотов, привязка к ноде.
- **Нода:** один Docker-образ с агентом + прокси-сервером. Агент регистрируется по токену, шлёт heartbeat и метрики, по запросу или по расписанию тянет список пользователей (логин, пароль, лимиты) и обновляет конфиг прокси (3proxy/gost).

## 3. Модель данных (концептуально)

### 3.1. Нода прокси (ProxyNode)

| Поле | Тип | Описание |
|------|-----|----------|
| id | cuid | |
| name | string | Название (админ задаёт при создании или нода при регистрации) |
| token | string, unique | Токен для регистрации и heartbeat (генерируется при «Добавить прокси») |
| status | enum | ONLINE, OFFLINE, DISABLED |
| lastSeenAt | DateTime | Последний успешный heartbeat |
| publicHost | string? | IP или домен ноды (из первого регистрации или из heartbeat) |
| socksPort | int | Порт SOCKS5 (например 1080) |
| httpPort | int | Порт HTTP-прокси (например 8080) |
| capacity | int? | Макс. слотов на ноде (опционально) |
| currentConnections | int | Текущее кол-во соединений (из heartbeat) |
| trafficInBytes | bigint | Входящий трафик (из heartbeat, накопительно или сброс) |
| trafficOutBytes | bigint | Исходящий трафик |
| metadata | json? | Доп. данные от агента (CPU, RAM, версия) |
| createdAt, updatedAt | DateTime | |

### 3.2. Категория прокси (ProxyCategory)

| Поле | Тип | Описание |
|------|-----|----------|
| id | cuid | |
| name | string | Например «Прокси РФ», «Прокси EU» |
| sortOrder | int | Порядок в списке |

### 3.3. Тариф прокси (ProxyTariff)

| Поле | Тип | Описание |
|------|-----|----------|
| id | cuid | |
| categoryId | FK → ProxyCategory | |
| name | string | Например «1 прокси 30 дней» |
| proxyCount | int | Кол-во прокси (слотов) в пакете (1, 5, 10…) |
| durationDays | int | Срок действия в днях |
| trafficLimitBytes | bigint? | Лимит трафика на один слот (null = без лимита) |
| connectionLimit | int? | Макс. одновременных подключений на слот |
| price | float | |
| currency | string | |
| sortOrder | int | |

### 3.4. Слот прокси (ProxySlot)

Один купленный «прокси» = один слот (один логин:пароль на одной ноде).

| Поле | Тип | Описание |
|------|-----|----------|
| id | cuid | |
| nodeId | FK → ProxyNode | На какой ноде выдан |
| clientId | FK → Client | Владелец |
| login | string | Уникальный в рамках ноды (например uuid или clientId_slotIndex) |
| passwordHash | string? | Или хранить пароль в открытом виде для выдачи юзеру (по аналогии с VPN-конфигами) — проще выдача. Лучше хранить plain password в зашифрованном виде или plain для выдачи. |
| expiresAt | DateTime | Конец срока |
| trafficLimitBytes | bigint? | Лимит на слот |
| connectionLimit | int? | |
| trafficUsedBytes | bigint | Фактический трафик (обновляется из heartbeat ноды) |
| currentConnections | int | Текущие соединения (из heartbeat) |
| status | enum | ACTIVE, EXPIRED, REVOKED |
| createdAt, updatedAt | DateTime | |

Уникальность: `(nodeId, login)` unique.

### 3.5. Оплата прокси

**Вариант A:** Отдельная таблица `ProxyPayment` (orderId, clientId, proxyTariffId, amount, currency, status, paidAt) и при статусе PAID создаём N слотов (N = proxyCount из тарифа).

**Вариант B:** Расширить существующую `Payment`: добавить `proxyTariffId`, при webhook PAID проверять — если есть proxyTariffId, создаём ProxySlot(ы) вместо/вместе с активации VPN-тарифа.

Рекомендация: **Вариант B** — меньше дублирования, единый поток оплаты (balance, Platega, YooMoney, YooKassa). В Payment добавить `proxyTariffId` (nullable). После успешной оплаты: если `proxyTariffId` задан — создать `proxyCount` слотов, распределить по нодам (round-robin или по наименьшей загрузке).

## 4. Сценарии

### 4.1. Админ: добавление прокси-ноды

1. В админке раздел «Прокси» → «Ноды» → кнопка «Добавить прокси».
2. Ввод названия ноды (опционально) → генерация уникального `token`.
3. Показ готового блока:
   - `docker-compose.yml` с переменными `STEALTHNET_API_URL`, `PROXY_NODE_TOKEN`, и при необходимости порты.
   - Краткая инструкция: скопировать, вставить на сервер, `docker compose up -d`.
4. После запуска контейнера на сервере админа:
   - Агент при старте отправляет `POST /api/proxy-nodes/register` с телом `{ "token": "...", "socksPort": 1080, "httpPort": 8080, "name": "..." }`.
   - API проверяет токен, создаёт или обновляет запись ProxyNode (status = ONLINE, lastSeenAt = now, publicHost из req.ip или заголовка).
   - Ответ: `{ "nodeId": "...", "pollIntervalSec": 60 }` (интервал heartbeat).
5. В списке нод админ видит новую ноду (онлайн, 0 слотов, 0 трафика).

### 4.2. Админ: heartbeat и метрики

1. Агент раз в N секунд: `POST /api/proxy-nodes/:id/heartbeat` с телом:
   - `connections` — общее кол-во соединений,
   - `trafficIn`, `trafficOut` — опционально общий трафик ноды,
   - `slots`: `[ { "slotId": "...", "trafficUsed": 0, "connections": 0 }, ... ]` — по каждому слоту на этой ноде (агент знает slotId из последнего ответа getSlots).
2. API обновляет ProxyNode (lastSeenAt, currentConnections, traffic*), обновляет ProxySlot (trafficUsedBytes, currentConnections).
3. Если heartbeat не приходит дольше порога (например 5 минут) — помечать ноду OFFLINE.

### 4.3. Админ: категории и тарифы прокси

1. «Категории прокси»: CRUD по аналогии с категориями тарифов VPN (название, порядок).
2. «Тарифы прокси»: внутри категории — название, кол-во прокси (proxyCount), срок (durationDays), лимит трафика на слот, лимит подключений на слот, цена, валюта.
3. Список тарифов в админке с возможностью редактирования и отключения (например флаг `enabled`).

### 4.4. Админ: просмотр нод и слотов

1. Список нод: таблица (название, статус, lastSeenAt, кол-во слотов, текущие соединения, трафик).
2. Клик по ноде → детали: список слотов на этой ноде (клиент, логин, срок, трафик использовано/лимит, соединения).
3. Опционально: графики по трафику/нагрузке по времени (позже).

### 4.5. Клиент: покупка прокси в боте

1. В боте новый пункт меню «Прокси» (или подраздел в «Тарифы»/отдельная кнопка).
2. Показ категорий и тарифов (как у VPN-тарифов): «1 прокси 30 дней — 100₽», «5 прокси 30 дней — 400₽» и т.д.
3. Выбор способа оплаты: баланс, Platega, ЮMoney, ЮKassa (аналогично тарифам).
4. После успешной оплаты:
   - Бэкенд создаёт `proxyCount` слотов: выбирает ноды (только ONLINE), распределяет слоты (по очереди или по минимальной загрузке), генерирует логин/пароль на каждый слот, записывает в ProxySlot.
   - В боте (и при необходимости по email/в кабинете) выдать список строк в формате:
     - `socks5://login:password@host:port`
     - `http://login:password@host:port`
     - Или текстом: `protocol:login:password:host:port` для копирования.
5. Раздел «Мои прокси» в боте/кабинете: список активных слотов с теми же строками, срок окончания, использовано трафика / лимит.

### 4.6. Выдача формата protocol:login:pass:ip:port

- Для каждого слота: `host` и `port` берутся из ProxyNode (publicHost, socksPort / httpPort).
- Два варианта выдачи:
  - Отдельно SOCKS5: `socks5://login:pass@host:1080` и HTTP: `http://login:pass@host:8080`.
  - Одна строка «универсальная»: можно выдать две строки (socks и http) в одном сообщении.
- Текстовый формат для копирования в старые клиенты: `socks5 login pass host port` и `http login pass host port` (или как принято в вашей аудитории).

## 5. Node Agent и прокси-сервер на ноде

### 5.1. Задачи агента

1. При старте: зарегистрироваться по токену (`POST /api/proxy-nodes/register`).
2. Периодически: heartbeat с метриками (`POST /api/proxy-nodes/:id/heartbeat`).
3. Периодически или по long-poll: запрос списка слотов для этой ноды (`GET /api/proxy-nodes/:id/slots`) — логин, пароль (plain), expiry, trafficLimit, connectionLimit.
4. По списку слотов генерировать конфиг для 3proxy (или gost) и перезагружать прокси-сервис.
5. Собирать с прокси-сервера статистику по пользователям (трафик, соединения) и отправлять в heartbeat.

### 5.2. Выбор прокси-сервера

- **3proxy:** конфиг-файл, поддержка auth, bandwidth limit, maxconn на пользователя. Минус: перезагрузка конфига при изменении списка пользователей (но это решаемо скриптом).
- **gost:** гибкий, auth, лимиты, можно управлять через API — удобнее для динамического списка пользователей.
- **Dante (sockd):** в основном SOCKS, нужен отдельный HTTP (например tinyproxy или nginx proxy). Сложнее комбинировать.

Рекомендация для первого этапа: **3proxy** — простой, стабильный, хорошо документирован. Агент при обновлении списка слотов переписывает конфиг и делает `kill -HUP` или перезапуск 3proxy.

### 5.3. Структура Docker-образа ноды

- Базовый образ: node:20-alpine или debian.
- Установка 3proxy (или gost).
- Скрипт агента (Node.js или Go): регистрация, heartbeat, загрузка слотов, генерация конфига 3proxy, перезапуск 3proxy.
- Entrypoint: запуск агента и 3proxy (supervisord или два процесса в одном контейнере).
- Переменные окружения: `STEALTHNET_API_URL`, `PROXY_NODE_TOKEN`, опционально `SOCKS_PORT`, `HTTP_PORT`.

### 5.4. API для агента (защита)

- Все запросы агента: заголовок `Authorization: Bearer <node_token>` или `X-Proxy-Node-Token: <token>`.
- Регистрация по токену: только если токен есть в БД (заранее создан админом при «Добавить прокси»). После первой регистрации можно привязать nodeId к токену и дальше требовать nodeId + тот же токен в heartbeat/getSlots.

## 6. Безопасность и ограничения

- Токен ноды хранить в БД в виде хэша или как есть (если панель недоступна снаружи, риск ниже). Предпочтительно хэш + соль, при регистрации сравнивать переданный токен с хэшем.
- Выдача слотов только на ноды со статусом ONLINE и не DISABLED.
- Пароли слотов: генерировать криптостойкие (crypto.randomBytes). В API для агента отдавать пароль в открытом виде только для своего nodeId (агент уже доверенный).
- Rate limit на регистрацию и heartbeat по IP/токену.
- При отзыве доступа: статус слота REVOKED, агент при следующем getSlots перестаёт выдавать этот логин в конфиг — 3proxy перезагружается без этого пользователя.

## 7. Этапы реализации

### Фаза 1: Бэкенд и админка (ноды, без продаж)

1. Миграции: таблицы ProxyNode, ProxyCategory, ProxyTariff, ProxySlot; в Payment добавить `proxyTariffId`.
2. API: CRUD нод (по токену создаётся запись «ожидание регистрации»), генерация токена при «Добавить прокси».
3. API для агента: `POST /api/proxy-nodes/register`, `POST /api/proxy-nodes/:id/heartbeat`, `GET /api/proxy-nodes/:id/slots` (возвращает логин/пароль, лимиты для каждого слота этой ноды).
4. Админка: раздел «Прокси» → вкладка «Ноды»: кнопка «Добавить прокси», вывод docker-compose + инструкция; список нод (статус, lastSeen, метрики). Детали ноды — список слотов.
5. Контейнер-агент + 3proxy: образ, скрипт регистрации/heartbeat/getSlots, генерация конфига 3proxy, тест на стенде.

### Фаза 2: Категории, тарифы, оплата и слоты

1. Админка: CRUD категорий и тарифов прокси (proxyCount, durationDays, trafficLimit, connectionLimit, price).
2. Backend: при оплате (balance или webhook Platega/YooMoney/YooKassa) проверка `proxyTariffId` → создание proxyCount слотов, выбор нод (ONLINE), распределение, генерация login/password.
3. Выдача в боте: после оплаты отправить клиенту сообщение со строками socks5 и http для каждого слота.
4. Бот: раздел «Прокси» — список тарифов, выбор и оплата (аналогично тарифам VPN); раздел «Мои прокси» — активные слоты и строки доступа.

### Фаза 3: Кабинет и доработки

1. Кабинет (сайт + Mini App): раздел «Прокси» — те же тарифы и «Мои прокси» с копированием строк.
2. Обновление трафика/подключений по heartbeat от ноды; отображение в админке (по ноде и по слоту) и у клиента («использовано X из Y ГБ»).
3. Автоматическое отключение слота при истечении срока или превышении лимита трафика; при необходимости — уведомление в боте/email.
4. Опционально: рефералы с продаж прокси (как для VPN); промокоды на прокси-тарифы.

### Фаза 4 (опционально)

- Резервирование нод по гео/тегам (ноды с тегом «EU» — тариф «Прокси EU» привязывается только к таким нодам).
- Графики нагрузки и трафика по нодам в админке.
- Экспорт списка слотов (CSV) для админа.

## 8. Краткий чеклист по формату выдачи

- Один слот = один логин:пароль на одной ноде.
- Выдаём клиенту:
  - `socks5://login:password@host:socksPort`
  - `http://login:password@host:httpPort`
- При необходимости добавить текстовую строку: `protocol:login:password:host:port` для каждого протокола.
- В «Мои прокси» показывать срок действия, лимит трафика (если задан), использовано трафика, лимит подключений.
